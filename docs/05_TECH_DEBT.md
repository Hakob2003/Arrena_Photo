# 💸 КАТАЛОГ ТЕХНИЧЕСКОГО ДОЛГА (Tech Debt Accounting)

Этот документ инвентаризирует технический долг проекта Arrena Photo. Технический долг здесь рассматривается не как "плохой код", а как финансовый и архитектурный риск (Бизнес-метрика). Каждая проблема имеет четкий Cost of Delay (Цена промедления).

Данный реестр обновляется архитектурной командой ежемесячно.

---

## 1. База Данных и Хранение Медиа

### 🚨 TD-DB-001: Хранение Base64 в PostgreSQL
**Описание:** Все финальные и промежуточные изображения сохраняются в таблицы БД (`Generation`, `Template`) в формате длинных строк Base64 (в полях `TEXT` или `BYTEA`).
**Источник:** Быстрый запуск MVP (избегание настройки AWS S3 и CORS политик).
**Влияние:** Критическое раздувание объема базы данных (каждое изображение занимает до 5МБ в базе, вытесняя кэши `shared_buffers`). Выполнение `SELECT` запросов замедляется в 100 раз. Невозможность быстрых бэкапов.
**Стоимость исправления:** Высокая (~40 часов инженера). Требует настройки S3, переписывания Воркера на AWS SDK, фоновой миграции миллионов строк.
**Стоимость игнорирования:** Fatal (Катастрофическая). При масштабировании (Scale) сервер БД упадет из-за исчерпания IOPS (дискового ввода-вывода). Хранение на AWS RDS обходится в 10 раз дороже, чем в S3.
**Приоритет:** P0 (Критический)
**ROI исправления:** Снижение счетов за инфраструктуру БД на 85%, ускорение API в 20 раз, гарантия восстановления из бекапа (RTO) за 5 минут.

### 🚨 TD-DB-002: Отсутствие Connection Pooler'а (PgBouncer)
**Описание:** Каждый под Node.js открывает собственные коннекты (connection pool) напрямую к PostgreSQL через Prisma.
**Источник:** Стандартная настройка Prisma без учета Cloud Native (Kubernetes) специфики.
**Влияние:** При всплеске трафика (наплыв юзеров) K8s поднимет 50 новых подов, каждый из которых откроет по 10 соединений. Лимит PostgreSQL (`max_connections=100`) будет мгновенно исчерпан. База ответит `FATAL: sorry, too many clients already`.
**Стоимость исправления:** Низкая (~5 часов). Развертывание PgBouncer (Transaction mode) и смена URL в Prisma.
**Стоимость игнорирования:** Даунтайм API при малейшем Spike'е трафика.
**Приоритет:** P1 (Высокий)
**ROI исправления:** Стабилизация базы данных, возможность Horizontal Pod Autoscaling (HPA) бэкенда до 1000 подов.

## 2. Архитектура Бэкенда (Монолит)

### 🚨 TD-ARCH-001: Синхронный Монолит (API + Worker)
**Описание:** Бизнес-логика (Rest API) и тяжелые вычислительные задачи (генерации ИИ, скачивание картинок) работают внутри одного процесса NestJS.
**Источник:** Монорепозиторный MVP без очереди задач.
**Влияние:** Node.js однопоточен (Event Loop). Обработка огромных Base64 Payload'ов (или ожидание I/O) блокирует поток. Обычные пользователи не могут даже залогиниться, пока кто-то другой генерирует картинку. Latency вырастает до 5000ms+.
**Стоимость исправления:** Средняя (~25 часов). Разделение на `apps/backend-api` и `apps/worker-ai`, внедрение BullMQ (Redis).
**Стоимость игнорирования:** Невозможность независимого масштабирования. Отвратительный UX из-за случайных зависаний приложения.
**Приоритет:** P0 (Критический)
**ROI исправления:** Идеальная отзывчивость API (P99 < 100ms), дешевое и эффективное масштабирование Воркеров (Spot Instances).

### 🚨 TD-SEC-001: Глобальный `any` и отсутствие DTO валидации
**Описание:** Запросы и ответы типизированы как `any` или не валидируются на входе (отсутствует `class-validator`).
**Источник:** Быстрое прототипирование.
**Влияние:** Уязвимость *Mass Assignment* (пользователь может послать `isAdmin: true` в JSON, и база это сохранит). Постоянные баги Frontend-Backend интеграции из-за измененного контракта.
**Стоимость исправления:** Очень высокая (~50 часов рутины). Замена всех Payload'ов на Zod/Class DTO.
**Стоимость игнорирования:** Взлом приложения, постоянные Runtime ошибки у клиентов.
**Приоритет:** P1 (Высокий)
**ROI исправления:** Нулевая терпимость к багам интеграции, автоматическая генерация Swagger, невозможность взлома через подделку JSON.

## 3. Архитектура Фронтенда (Next.js)

### 🚨 TD-FRONT-001: Глобальный `'use client'` в Корневом Layout
**Описание:** Проект использует Next.js 14 App Router, но `app/layout.tsx` обернут в `'use client'`, что превращает весь проект в SPA (Single Page Application).
**Источник:** Неопытность при работе с архитектурой React Server Components (RSC) или интеграция старого Redux провайдера.
**Влияние:** Отсутствие SSR. Роботы Google видят пустую страницу. Нулевой шанс на органический SEO-трафик. JS бандл весит мегабайты, убивая Time-to-Interactive (TTI) на мобильных.
**Стоимость исправления:** Высокая (~40 часов). Спуск стейта "вниз" по дереву, внедрение Server Components.
**Стоимость игнорирования:** Проект будет жить только на платном трафике (Ads).
**Приоритет:** P0 (Бизнес-критический)
**ROI исправления:** 100 баллов Lighthouse, бесплатный органический SEO трафик, мгновенная загрузка страниц.

### 🚨 TD-FRONT-002: Отсутствие LCP Оптимизации Изображений
**Описание:** Сгенерированные фотографии выводятся обычными тегами `<img>` без атрибутов `priority`, `sizes`, или формата `WebP`.
**Источник:** Пренебрежение компонентом `next/image`.
**Влияние:** Плохие Core Web Vitals. Google пессимизирует сайт в выдаче за долгую отрисовку главной картинки (Largest Contentful Paint > 4s).
**Стоимость исправления:** Низкая (~8 часов). Внедрение `next/image` с Cloudflare/S3 Image Loader.
**Стоимость игнорирования:** Скрытый штраф от Google, отток пользователей с плохим интернетом (3G/4G Mobile).
**Приоритет:** P2 (Средний)
**ROI исправления:** Плавный и быстрый UX, соответствие стандартам Google Core Web Vitals.

## 4. Секретность и Инфраструктура

### 🚨 TD-INFRA-001: Hardcoded Secrets и Fallbacks
**Описание:** В коде присутствуют строки вида `process.env.JWT_SECRET || 'super_secret'`.
**Источник:** Локальная разработка без `.env` файлов.
**Влияние:** В Production, если DevOps забудет прописать ENV, приложение запустится со статичным ключом, известным из исходного кода. Это позволит хакерам генерировать админские JWT токены.
**Стоимость исправления:** Минимальная (~2 часа). Паттерн "Fail Fast" с использованием `ConfigService.getOrThrow()`.
**Стоимость игнорирования:** Тотальный Data Breach, уничтожение бизнеса.
**Приоритет:** P0 (Немедленный)
**ROI исправления:** Прохождение комплаенса и безопасность данных.

### 🚨 TD-INFRA-002: Отсутствие CI/CD Блокировок (Lint/Test)
**Описание:** Ветка `main` не защищена (Branch Protection Rules). PR можно влить с красными тестами или ошибками линтера.
**Источник:** Ненастроенные GitHub Actions.
**Влияние:** Нерабочий код регулярно попадает в Production (Broken Main). Команда тратит время на тушение пожаров (Hotfixes).
**Стоимость исправления:** Средняя (~10 часов). Настройка Github Actions (ESLint, Prettier, Jest, Prisma Generate).
**Стоимость игнорирования:** Падение скорости разработки (Velocity) из-за страха релизов.
**Приоритет:** P1 (Высокий)
**ROI исправления:** Надежность деплоя, гарантия того, что в `main` попадает только рабочий код.
