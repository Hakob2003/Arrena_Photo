# 🚀 УЛЬТИМАТИВНЫЙ PRODUCTION CHECKLIST (Enterprise Standard)

Этот чеклист обязателен к выполнению перед каждым деплоем в Production. Пропуски в разделах Security, Database или DevOps означают жесткий запрет на релиз (NO-GO).

---

## 🔒 Security (Безопасность)

- [ ] Убедиться, что эндпоинт `/setup-admin` (бэкдор) полностью вычищен из кодовой базы (проверка через `grep`).
- [ ] Переменные окружения (`JWT_SECRET`, `STRIPE_SECRET_KEY`, `OPENAI_API_KEY`) проверяются при старте `main.ts` через валидатор Zod (никаких Fallbacks вида `|| 'secret'`).
- [ ] В `next.config.js` включен заголовок `Strict-Transport-Security: max-age=31536000; includeSubDomains; preload`.
- [ ] Заголовок `X-Frame-Options: DENY` настроен глобально (защита от Clickjacking).
- [ ] Заголовок `X-Content-Type-Options: nosniff` настроен глобально.
- [ ] Content Security Policy (CSP) не содержит `unsafe-inline` для JavaScript (допускается только для стилей Tailwind).
- [ ] Все публичные API-роуты закрыты глобальным Rate Limiting (100 req/min).
- [ ] Эндпоинты `/auth/login` и `/auth/register` защищены жестким лимитом (5 req/min) (защита от Brute Force).
- [ ] Лимиты Rate Limiting хранятся в Redis (shared state), а не в памяти отдельного пода.
- [ ] Пароли в базе данных хешируются с использованием `bcrypt` (salt rounds >= 10).
- [ ] Внедрен Role-Based Access Control (RBAC). Админка защищена строгим декоратором `@Roles('ADMIN')`.
- [ ] Глобально отключен CORS для неизвестных Origin (`cors: { origin: 'https://arrena.ai' }`).
- [ ] Скрыта версия фреймворка (убран заголовок `X-Powered-By: Express` и мета-тег `Next.js`).
- [ ] JWT токены имеют короткий срок жизни (Access Token < 15 минут).
- [ ] Refresh Tokens хранятся ИСКЛЮЧИТЕЛЬНО в `HttpOnly, Secure, SameSite=Strict` cookies.
- [ ] Секрет для генерации JWT генерируется криптографически стойким генератором (минимум 256 бит).
- [ ] Отключен Directory Listing на всех статических серверах (Nginx / S3).
- [ ] Настроена фильтрация загружаемых файлов (Avatar Upload пропускает только `image/jpeg`, `image/png`, `image/webp`).
- [ ] Проверка размера загружаемого файла (Max 5MB) происходит ДО сохранения файла на диск/в S3.
- [ ] Файлы, загруженные пользователем, сканируются на наличие вредоносного кода (например, через ClamAV), если они публично доступны.

## 🏗️ Infrastructure (Инфраструктура)

- [ ] Вся инфраструктура разворачивается с помощью IaC (Terraform). Ручные изменения (ClickOps) через веб-консоль провайдера запрещены.
- [ ] Настроена приватная сеть (VPC). База данных PostgreSQL и Redis находятся в Private Subnet и не имеют публичных IP адресов.
- [ ] Для доступа администраторов к ресурсам VPC используется Bastion Host или AWS Systems Manager Session Manager.
- [ ] Использованы Least Privilege IAM роли (Например, Worker Pod имеет роль, разрешающую только `s3:PutObject` для конкретного бакета, а не `AdministratorAccess`).
- [ ] Сертификаты SSL/TLS обновляются автоматически за 30 дней до истечения (Let's Encrypt / AWS ACM).
- [ ] Load Balancer или Ingress Controller (NGINX/Traefik) пробрасывает реальный клиентский IP в заголовок `X-Forwarded-For`.
- [ ] Настроена WAF (Web Application Firewall) на уровне CDN (Cloudflare) для защиты от Layer 7 атак (боты, SQLi, XSS).
- [ ] Настроен лимит размера тела запроса на уровне Ingress/NGINX (`client_max_body_size 10M`), чтобы предотвратить OOM.
- [ ] Настроена изоляция ресурсов Kubernetes (Resource Quotas и LimitRanges) для предотвращения ситуации "Noisy Neighbor".
- [ ] Узлы кластера (Nodes) находятся в разных зонах доступности (Multi-AZ) для защиты от дата-центровых аварий.
- [ ] Настроены группы безопасности (Security Groups): открыты только порты 80 и 443 для интернета, порт 5432 открыт только для внутренней подсети.
- [ ] Регулярно запускается автоматический сканер уязвимостей в Docker-образах (Trivy / AWS Inspector).
- [ ] Инфраструктура стейджинга (Staging) полностью идентична Production, но смасштабирована вниз.
- [ ] Выделен отдельный IAM аккаунт/роль для CI/CD системы с ограниченными правами.

## ⚙️ Backend (Бэкенд)

- [ ] Проект разделен на независимые монорепозиторные пакеты: `backend-api` (HTTP) и `worker-ai` (BullMQ Processor).
- [ ] В `tsconfig.json` жестко включен `strict: true` (никаких `implicit any`).
- [ ] В ESLint включено правило `@typescript-eslint/no-explicit-any` со статусом `error`.
- [ ] В NestJS включен глобальный `ValidationPipe` с настройками `whitelist: true`, `forbidNonWhitelisted: true`, `transform: true`.
- [ ] Внедрена обработка Graceful Shutdown (`app.enableShutdownHooks()`), чтобы при SIGTERM сервер закрывал Prisma коннекты и дописывал логи.
- [ ] Все необработанные ошибки (500 Internal Server Error) проходят через глобальный `ExceptionFilter` и не сливают `StackTrace` клиенту.
- [ ] Входящие данные (Payloads) типизируются DTO-классами с декораторами `@IsString`, `@IsNumber`, `@IsOptional`.
- [ ] Уязвимые к атакам переполнения массивы проверяются через `@ArrayMaxSize(100)`.
- [ ] Внедрены ключи идемпотентности (Idempotency Keys) для всех POST/PUT запросов, связанных с биллингом и транзакциями.
- [ ] Бэкенд возвращает стандартизированный формат ошибок (RFC 7807 Problem Details for HTTP APIs).
- [ ] Для всех длительных задач (генерация изображений, экспорт данных) используется паттерн асинхронного ответа (`202 Accepted` + `Location` заголовок для поллинга статуса).
- [ ] Логика работы с внешними провайдерами (OpenAI, Stripe) обернута в паттерн Retry с Exponential Backoff.
- [ ] Внедрен паттерн Circuit Breaker для предотвращения каскадных падений при отказе внешних сервисов (AI, Billing).
- [ ] Не существует циклических зависимостей (Circular Dependencies) между NestJS модулями.
- [ ] API эндпоинты версионированы (например, `/v1/generations`).

## ⚛️ Frontend (Фронтенд)

- [ ] Корневой `app/layout.tsx` является Server Component (отсутствует глобальный `'use client'`).
- [ ] Критический JS-бандл (First Load JS) главной страницы не превышает 150 KB (gzipped).
- [ ] Исключены утечки памяти в React `useEffect`: все подписки на вебсокеты, `setInterval` и обработчики событий очищаются (`return () => cleanup()`).
- [ ] Ошибки роутинга элегантно обрабатываются файлами `error.tsx` и `not-found.tsx` на каждом уровне навигации.
- [ ] Базовые UI-компоненты (модалки, дропдауны, кнопки) изолированы в библиотеке `@arrena/ui-kit` (Radix UI) и не содержат бизнес-логики.
- [ ] Используется оптимистичный UI (Optimistic Updates) через React Query / SWR для мгновенной обратной связи при действиях пользователя (лайк, удаление).
- [ ] Включен React Strict Mode (`reactStrictMode: true` в `next.config.js`) для выявления скрытых проблем рендеринга.
- [ ] Для всех тяжелых компонентов (сложные графики, 3D) используется ленивая загрузка (`next/dynamic`).
- [ ] Переменные окружения, которые не должны попасть в клиентский бандл, НЕ имеют префикса `NEXT_PUBLIC_`.
- [ ] Чувствительные данные (баланс, токены) не кэшируются агрессивно сервером (fetch с `cache: 'no-store'`).
- [ ] Состояние приложения, не требующее реактивности, вынесено из Zustand/Redux обратно в локальный стейт или в URL (Query Params).
- [ ] Все ссылки на внешние ресурсы используют `target="_blank" rel="noopener noreferrer"`.
- [ ] Мобильное меню (Burger Menu) автоматически закрывается при смене роута (pathname change).
- [ ] Защищенные роуты (Protected Routes) проверяют сессию пользователя на сервере (Middleware / Server Component), а не на клиенте, во избежание мелькания закрытого контента (FOUC).

## 🗄️ Database (База Данных)

- [ ] Бинарные файлы и Base64-строки категорически НЕ хранятся в таблицах PostgreSQL (перенесены в S3/MinIO).
- [ ] Внедрен Connection Pooler (PgBouncer) в режиме `transaction mode` перед БД.
- [ ] Prisma Client подключается с флагом `?pgbouncer=true`.
- [ ] Лимит подключений пула в Node.js установлен на минимальное значение (`connection_limit=3-5`), чтобы сэкономить ресурсы PgBouncer.
- [ ] Последняя миграция Prisma применена в Production ДО старта нового кода (`prisma migrate deploy`).
- [ ] Таблицы `Generation`, `Template`, `User` имеют индексы (B-Tree) по ключам, используемым в WHERE, ORDER BY (например, `userId`, `createdAt`).
- [ ] Не существует неиспользуемых индексов, замедляющих INSERT операции.
- [ ] Настроен агрессивный `autovacuum` для предотвращения Bloat (накопления мертвых строк), особенно для таблиц сессий и очередей задач.
- [ ] Внедрено Софт-удаление (Soft Delete) через поле `deletedAt`. Ни одна значимая бизнес-запись не удаляется физически (`DELETE FROM`).
- [ ] Уникальные ограничения (Unique Constraints) настроены на уровне базы данных, а не только валидируются в коде (например, для `email`).
- [ ] Foreign Keys (внешние ключи) настроены с корректными правилами `ON DELETE RESTRICT` или `ON DELETE CASCADE`, чтобы избежать нарушения ссылочной целостности (Orphan records).
- [ ] Для больших таблиц (> 10M строк) предусмотрено партицирование по дате (Partitioning by RANGE `createdAt`).
- [ ] Включено логирование долгих запросов (Slow Query Log, `log_min_duration_statement = 500ms`).
- [ ] Настроен механизм потоковой репликации (Streaming Replication) для обеспечения High Availability.


## 🚢 DevOps (DevOps и Контейнеризация)

- [ ] Docker-образы собираются через Multi-stage builds. В финальный образ не попадают исходники Typescript и `devDependencies` (размер образа < 300MB).
- [ ] Финальный Docker-образ базируется на легковесном дистрибутиве (`node:18-alpine` или `distroless`).
- [ ] Контейнеры запускаются из-под непривилегированного пользователя (инструкция `USER node`, а не `root`).
- [ ] GitHub Actions пайплайн имеет жесткую блокировку: проверка линтеров (`pnpm lint`) и типов (`pnpm typecheck`) выполняется перед запуском билда.
- [ ] В Kubernetes манифестах (или Docker Compose) настроены `readinessProbe` (под готов принимать трафик) и `livenessProbe` (под жив).
- [ ] Настроены Resource Requests и Resource Limits для всех подов (CPU, Memory), чтобы предотвратить Out Of Memory (OOM) на ноде K8s.
- [ ] Переменные окружения доставляются в контейнеры через защищенные хранилища секретов (Kubernetes Secrets, AWS Secrets Manager, HashiCorp Vault), а не хардкодятся в манифестах.
- [ ] Настроен Horizontal Pod Autoscaler (HPA) для масштабирования API подов в зависимости от утилизации CPU (> 70%).
- [ ] Настроен KEDA (Kubernetes Event-driven Autoscaling) для автоматического масштабирования Worker подов в зависимости от длины очереди BullMQ в Redis.
- [ ] Образы заливаются в приватный Container Registry (ECR, GCR) с тегированием по Git Commit SHA (тег `latest` не используется в Production).
- [ ] Резервные копии инфраструктурного кода (Terraform state) хранятся в удаленном зашифрованном бакете с блокировкой состояния (State Locking) через DynamoDB.

## ⚡ Performance (Производительность)

- [ ] P99 Latency REST API не превышает 100ms для критичных маршрутов (проверка через нагрузочное тестирование `k6` или `artillery`).
- [ ] Полностью исключены блокировки Event Loop. Все операции парсинга JSON > 1MB или обработки изображений переданы Worker'ам или выполняются в `Worker Threads`.
- [ ] Включено кэширование публичных ответов (Redis) для справочников и эндпоинтов, не требующих аутентификации.
- [ ] Для защиты кэша от "Cache Stampede" реализован механизм рандомизации TTL (Jitter) или лок на обновление.
- [ ] Статический контент (шрифты, JS-бандлы, CSS) отдается с заголовками `Cache-Control: public, max-age=31536000, immutable`.
- [ ] Статика раздается через распределенную сеть доставки контента (CDN).
- [ ] Cloudflare Image Optimization (или S3 Image Loader) используется для динамического изменения размера (resize) и сжатия картинок на лету (форматы WebP/AVIF).
- [ ] БД-запросы оптимизированы: отсутствуют проблемы N+1. Prisma Client использует `include` и `select` для выборок связанных сущностей одним запросом.
- [ ] Реализована пагинация (Keyset/Cursor-based Pagination) для списков (галерей) вместо медленной Offset-based пагинации (`SKIP / TAKE`).
- [ ] Огромные SQL запросы (> 1 миллиона строк) не используют оператор `COUNT(*)`, так как это приводит к полному сканированию таблицы в PostgreSQL.

## 🧪 Testing (Тестирование и Контроль качества)

- [ ] Код бизнес-логики (Billing, Validation, Auth) покрыт Unit-тестами (Jest / Vitest). Минимальный порог Coverage > 70%.
- [ ] DTO классы протестированы на негативные сценарии (переполнение строк, передача неверных типов данных, SQL Injection).
- [ ] Интеграционные тесты (Supertest) проверяют реальное взаимодействие API с тестовой базой данных (Prisma + Testcontainers).
- [ ] В интеграционных тестах мокаются внешние зависимости (Stripe API, OpenAI API).

- [ ] Тесты выполняются параллельно (Parallel Execution) для минимизации времени ожидания CI-пайплайна (< 10 минут на весь цикл).
- [ ] Настроен Visual Regression Testing (Storybook + Chromatic) для компонентов UI-Kit, чтобы гарантировать отсутствие поехавшей верстки.
- [ ] Настроены тесты на проверку гонок данных (Race Conditions) в критичных участках биллинга.

## 🛫 Release (Релизный Процесс)

- [ ] Версионирование (Semantic Versioning) ведется строго по правилам `MAJOR.MINOR.PATCH` на основе коммитов (Conventional Commits).
- [ ] Сборка Релиз-кандидата (RC) автоматически публикуется на Staging окружении.
- [ ] Staging-окружение одобрено QA-инженером (или Tech Lead'ом) перед нажатием кнопки "Deploy to Production".
- [ ] Деплой происходит без даунтайма (Zero Downtime Deployment / Rolling Update K8s). Пользователи не получают 502/503 ошибки во время подмены подов.
- [ ] Новые миграции базы данных (Prisma Migrations) запускаются в виде `initContainer` или Job'а ДО того, как новый код начнет получать HTTP-трафик.
- [ ] Автоматически генерируется Changelog (`CHANGELOG.md`) и релиз-ноты в GitHub Releases.
- [ ] Оповещение об успешном деплое автоматически отправляется в командный Slack / Telegram канал.
- [ ] В случае выкатки "опасной" функциональности используется Feature Flags (LaunchDarkly или кастомное решение) для плавного включения (Canary Release).
- [ ] Команда DevOps/Backend находится в боевой готовности (on-call) во время окна деплоя.
- [ ] Деплой в пятницу вечером строго запрещен политикой компании (No Deploy Friday).

## 🔙 Rollback (Процедура Отката)

- [ ] Существует четко задокументированная, протестированная процедура отката (Rollback Procedure) в случае фатальных ошибок после релиза.
- [ ] Система оркестрации K8s / ArgoCD позволяет откатить состояние приложения к предыдущему `git tag` нажатием одной кнопки (или одной командой `helm rollback`).
- [ ] Разрушающие миграции базы данных (например, переименование или удаление столбца `DROP COLUMN`) запрещены в один этап.
- [ ] Паттерн "Expand and Contract": миграция проводится в 3 этапа. (1. Добавить новое поле. 2. Писать в оба поля, читать из нового. 3. Удалить старое поле). Это гарантирует, что старая версия кода будет работать с новой БД при Rollback'е.
- [ ] Перед каждым деплоем (особенно при изменении схемы БД) делается мгновенный Point-in-Time Recovery Snapshot или бекап (`pg_dump`).
- [ ] Настроен механизм "Feature Flag Kill Switch" — возможность выключить сломанную фичу в реальном времени, не откатывая весь релиз.
- [ ] Секреты (старые JWT ключи, пароли БД) не удаляются и не инвалидируются сразу же после их ротации, чтобы дать время на откат приложения.
- [ ] Инциденты с Rollback завершаются написанием Blameless Postmortem (документа с разбором причин, без поиска виноватых).

## 👁️ Observability (Наблюдаемость и Трейсинг)

- [ ] Внедрен Distributed Tracing (Sentry / Jaeger / OpenTelemetry). Заголовок `x-request-id` или `TraceId` передается по всему графу вызовов: от фронтенда к API, затем в Redis BullMQ, и затем к Worker-у.
- [ ] Инженер может в один клик по TraceId найти ВСЕ логи (nginx, nestjs, bullmq), связанные с конкретным запросом пользователя.
- [ ] Логи выводятся ИСКЛЮЧИТЕЛЬНО в формате JSON (Pino / Winston) для удобства парсинга лог-агентами (FluentBit / Logstash).
- [ ] "Человекочитаемые" цветные логи (NestJS default logger) жестко отключены в Production, так как они ломают парсеры.
- [ ] Логи маскируют PII данные (Personally Identifiable Information: Email, Password, Credit Cards, Auth Tokens).
- [ ] В Sentry настроено игнорирование спам-ошибок (404 Not Found) для экономии квоты.
- [ ] Пользовательские сессии связаны с ошибками в Sentry (к ошибке прикрепляется `userId` или `email`).
- [ ] При падении Worker-сервиса (Unhandeled Exception) стек-трейс автоматически захватывается и отправляется в Sentry до того, как процесс завершится (через хуки `process.on('uncaughtException')`).
- [ ] В логи пишется детальный контекст (например, `jobId`, `providerName`, `attemptCount`), а не просто текст ошибки.
- [ ] Логи агрегируются в централизованном хранилище (ElasticSearch / Datadog / Loki) с глубиной хранения не менее 14-30 дней для расследования инцидентов.
