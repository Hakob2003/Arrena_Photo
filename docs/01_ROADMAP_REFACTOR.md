# 🗺️ ROADMAP REFACTORING (Enterprise Master Plan)

Этот стратегический документ (Roadmap) описывает пошаговый процесс полного рефакторинга и трансформации монолитной кодовой базы MVP Arrena Photo в отказоустойчивую Cloud-Native SaaS-платформу уровня Enterprise.

> [!CAUTION]  
> Переход к следующему спринту **строго запрещен** до полного закрытия всех критериев завершения (DoD) текущего спринта. Игнорирование этого правила приведет к каскадным сбоям на Production и невозможности отката (Rollback).

---

## 🔒 Sprint 0 — Security (Фундамент Безопасности)

### Цель
Полностью закрыть критические уязвимости (OWASP Top 10), удалить тестовые бэкдоры, устранить хардкод секретов и выстроить базовый периметр безопасности (Zero Trust) для защиты данных пользователей.

### Ожидаемый результат
Система гарантированно защищена от несанкционированного доступа к административным роутам. Проект падает (Fail-Fast) при старте, если конфигурация среды (ENV) неполная или скомпрометирована. Атаки типа Brute Force и DDoS на уровне L7 не приводят к падению базы данных.

### Причины выполнения
В текущей MVP-версии (согласно Аудиту) существуют катастрофические уязвимости:
1. Эндпоинт `/setup-admin` позволяет любому анонимному HTTP-клиенту выдать себе роль администратора.
2. В файлах сервисов (например, `auth.service.ts`) жестко зашиты секреты по умолчанию (fallbacks: `process.env.JWT_SECRET || 'secret'`), что при ошибке деплоя откроет систему для подделки токенов.
3. Отсутствие Rate Limiting позволяет ботам генерировать десятки тысяч запросов в секунду, сжигая баланс LLM-провайдеров и убивая БД.

### Зависимости
Отсутствуют. Это абсолютный базис. Никакой рефакторинг архитектуры или фронтенда не имеет смысла, если система может быть взломана скрипт-кидди за 5 минут.

### Риски
- При агрессивном вычищении неиспользуемого кода (бэкдоров) существует риск случайного удаления легитимных зависимостей (Dead Code Elimination failure), что приведет к сбоям при сборке.
- Внедрение Rate Limiting может ошибочно заблокировать пользователей корпоративных сетей (NAT), если неверно настроено чтение заголовка `X-Forwarded-For` за Ingress контроллером.
- Паттерн Fail-Fast приведет к отказу в запуске на Staging-окружениях, если DevOps-команда не успеет обновить хранилища секретов.

### Влияние на проект
Закладывает криптографический и архитектурный фундамент. Позволяет системе пройти базовые аудиты информационной безопасности (SOC2 readiness) и предотвращает юридические риски, связанные с возможной утечкой PII (Personally Identifiable Information).

### Примерная длительность
1-2 недели (зависит от скорости согласования политик CSP и настройки Redis для Throttling-а).

### Критерии завершения (DoD)
- [ ] Физически удален код эндпоинта `/setup-admin` из `auth.controller.ts` и соответствующая логика из сервисов.
- [ ] Переменные `JWT_SECRET`, `STRIPE_KEY`, `OPENAI_API_KEY` валидируются Zod схемой в NestJS (`@nestjs/config`). Код содержит вызов `throw new Error()` при отсутствии переменных.
- [ ] Развернут Redis для распределенного хранения счетчиков Rate Limiting. Установлен лимит 100 req/min глобально и 5 req/min для `/login`.
- [ ] Внедрена подсистема Role-Based Access Control (RBAC). Создан `RolesGuard`, который на уровне NestJS контекста проверяет права пользователя для `/admin/*` путей.
- [ ] Фронтенд (Next.js) возвращает строгие HTTP-заголовки безопасности: HSTS, X-Content-Type-Options, CSP без `unsafe-inline` для скриптов.

## 🏗️ Sprint 1 — Infrastructure & Database Isolation

### Цель
Разорвать синхронный NestJS монолит на изолированные вычисляемые единицы (API и Worker). Прекратить токсичную практику хранения бинарных изображений внутри реляционной базы PostgreSQL.

### Ожидаемый результат
База данных "худеет" на 90% по объему хранимых данных. API-сервис начинает отвечать за миллисекунды, так как тяжелые I/O задачи (ожидание ответов ИИ, конвертация Base64 -> Buffer) перенесены в изолированные Worker-узлы.

### Причины выполнения
В текущей реализации запрос на генерацию картинки блокирует Node.js Event Loop. Каждое изображение (весом до 5МБ) пишется в таблицы `TEXT`, вытесняя ценные данные (индексы) из оперативной памяти PostgreSQL (`shared_buffers`). При нагрузке в 100 одновременных пользователей база данных исчерпает ресурсы ввода-вывода (IOPS) и упадет.

### Зависимости
Завершение **Sprint 0**. Защищенный периметр необходим для безопасной работы Worker-узлов с облачным хранилищем (IAM Roles).

### Риски
- Миграция сотен гигабайт Base64-строк из PostgreSQL в AWS S3 (объектное хранилище) в фоновом режиме может создать каскадную блокировку таблиц, если выполнять `SELECT` запросы без пагинации (Cursor-based).
- Разрыв монолита на микросервисы потребует внедрения Message Broker'а (BullMQ/Redis). При падении Redis (до настройки Persistent Storage) возможна потеря оплаченных пользовательских задач (Jobs).

### Влияние на проект
Абсолютное разблокирование горизонтального масштабирования (Horizontal Pod Autoscaling). Резкое снижение инфраструктурных затрат (AWS S3 обходится в десятки раз дешевле, чем AWS RDS хранилище того же объема). Увеличение отказоустойчивости.

### Примерная длительность
3-4 недели.

### Критерии завершения (DoD)
- [ ] Репозиторий переведен в режим PNPM Workspaces (`apps/backend-api`, `apps/worker-ai`, `packages/database`).
- [ ] Поднят микросервис `worker-ai` (NestJS без HTTP-сервера). Задачи на генерацию передаются из `backend-api` строго через очереди BullMQ.
- [ ] Настроен AWS SDK или S3-совместимый клиент. Все сгенерированные фото пишутся напрямую в объектное хранилище (используя Multipart Upload для крупных файлов).
- [ ] В `schema.prisma` добавлено поле `s3ImageUrl String?`. Написан и успешно запущен скрипт фоновой миграции Base64 -> S3 (без простоя системы).
- [ ] Настроен PgBouncer в режиме Transaction Pooling для защиты пула коннектов PostgreSQL от всплесков трафика.


## ⚙️ Sprint 2 — Backend & Type Safety

### Цель
Создать пуленепробиваемые контракты данных (DTO) между сервером и клиентом. Уничтожить использование `any` во всем проекте. Внедрить финансовую безопасность (Billing Engine) для предотвращения двойных списаний.

### Ожидаемый результат
Zero `any` policy во всем репозитории. Автогенерация Swagger-документации на основе DTO. Платежная подсистема, способная выдержать сетевые лаги без потери денег клиента (Chargeback prevention).

### Причины выполнения
Слабая типизация приводит к уязвимостям Mass Assignment: злоумышленник может отправить лишние поля (например, `isAdmin: true` или `balance: 9999`) в теле POST-запроса, и база данных это молча сохранит. Отсутствие ключей идемпотентности в интеграции Stripe означает, что если пользователь дважды быстро нажмет кнопку "Оплатить", с его карты спишется двойная сумма.

### Зависимости
Завершение **Sprint 1**. Для создания единого пакета `shared-types` требуется настроенная архитектура PNPM Workspaces.

### Риски
- Масштабный рефакторинг типов затронет сотни файлов фронтенда. Существует риск появления "красных" экранов (Type Errors) в неочевидных местах, если миграция будет проведена недостаточно аккуратно.
- Замена слабых валидаторов на жесткий `ValidationPipe(whitelist: true)` может отбросить легитимные данные от старых версий мобильных клиентов (обратная совместимость).

### Влияние на проект
Полная предсказуемость контрактов (DX - Developer Experience). Невозможность сломать фронтенд изменениями в базе данных (TypeScript "поймает" ошибку во время сборки). Автогенерация документации экономит сотни часов на онбординг новых разработчиков. Финансовая безупречность.

### Примерная длительность
3 недели (включая рутинную замену типов `any` на конкретные интерфейсы и схемы Zod).

### Критерии завершения (DoD)
- [ ] В `tsconfig.json` (корень и все пакеты) активирован флаг `strict: true` и `noImplicitAny: true`.
- [ ] Создан локальный пакет `@arrena/shared-types`, содержащий Zod-схемы ответов. Он импортируется как в Next.js, так и в NestJS.
- [ ] В NestJS подключен глобальный `ValidationPipe` с флагами `whitelist: true` и `forbidNonWhitelisted: true`.
- [ ] Во все операции биллинга (создание платежей, списание кредитов) внедрены уникальные ключи идемпотентности (Idempotency Keys формата UUIDv7).
- [ ] Реализован паттерн Compensating Transactions: если пользователь заказал генерацию, кредиты списались, но LLM провайдер ответил фатальной ошибкой (400 Bad Request), кредиты автоматически возвращаются на баланс пользователя.


## ⚛️ Sprint 3 — Frontend Architecture (App Router & SSR)

### Цель
Полная трансформация Frontend-приложения из "тяжелого" SPA в SEO-оптимизированный продукт на базе React Server Components (RSC). Снижение времени первой отрисовки (FCP/LCP) и перенос тяжеловесной логики на сервер.

### Ожидаемый результат
HTML страницы полностью рендерятся на сервере (Server-Side Rendering). Критический JS-бандл уменьшен вдвое. Поисковые системы (Google, Yandex) беспрепятственно индексируют страницы. 

### Причины выполнения
Использование глобальной директивы `'use client'` в корневом `app/layout.tsx` (часто из-за Redux или Theme-провайдеров) убивает главное преимущество Next.js. Весь сайт отправляется клиенту в виде пустого `<div id="root"></div>`, что уничтожает шансы на получение бесплатного органического трафика (SEO). Излишний размер JS бандла приводит к огромному Bounce Rate (показатель отказов) на медленных мобильных сетях.

### Зависимости
Завершение **Sprint 2**. Строгие Shared Types необходимы для безопасного получения (Data Fetching) данных в Server Components без дублирования интерфейсов.

### Риски
- Спуск глобального стейта (Zustand/Context) до уровня Leaf Components (листьев дерева) может потребовать значительного переписывания архитектуры состояний (State lifting / Prop drilling).
- Переход на Server Actions может вскрыть проблемы с авторизацией, если куки/заголовки передаются некорректно.
- Высокий риск появления трудноотлаживаемых Hydration Errors (несовпадение HTML сервера и клиента).

### Влияние на проект
Открывает дверь для неограниченного масштабирования бесплатного органического трафика. Делает пользовательский опыт (UX) плавным и отзывчивым даже на старых смартфонах.

### Примерная длительность
4 недели (самый объемный спринт по фронтенду).

### Критерии завершения (DoD)
- [ ] Физически удален `'use client'` из корневых Layouts (`app/layout.tsx`). Максимум компонентов переведены в Server Components.
- [ ] Состояние приложения изолировано: Zustand или Context API используются исключительно там, где нужна интерактивность (Island Architecture).
- [ ] Внедрен пакет `@arrena/shared-types` для валидации ответов (fetch returns) прямо на уровне Server Components.
- [ ] Статические картинки и фото пользователей отдаются ИСКЛЮЧИТЕЛЬНО через компонент `<Image>` из `next/image` с автоматической конвертацией в WebP/AVIF.
- [ ] Оптимистичные UI апдейты (Optimistic Updates) через SWR / React Query настроены для кнопок "Лайк" и "Удалить" — интерфейс реагирует мгновенно, не дожидаясь ответа сервера.


## 🚀 Sprint 4 — Performance & Caching

### Цель
Устранение всех узких мест производительности на слоях Базы Данных, API и Фронтенда. Подготовка системы к виральным всплескам трафика (Viral Spikes) и достижение 100 баллов Google Lighthouse.

### Ожидаемый результат
Платформа выдерживает одновременный наплыв в 5000+ RPS (Requests Per Second) на чтение справочников и профилей без деградации базы данных. P99 Latency REST API стабильно удерживается ниже 100ms. Frontend загружается до TTI (Time to Interactive) менее чем за 1.5 секунды.

### Причины выполнения
Медленный сервис (подвисания интерфейса, долгая загрузка фото) вызывает лавинообразный отток пользователей (Churn). Нехватка правильных B-Tree индексов в PostgreSQL приводит к полному сканированию таблиц (Seq Scan), что убивает процессор базы данных (100% CPU Utilization).

### Зависимости
Завершение **Sprint 3**. Оптимизировать фронтенд без внедрения Server Components невозможно. Кэширование на бэкенде требует жестких контрактов (Sprint 2).

### Риски
- Агрессивное кэширование (Redis/CDN) часто приводит к проблеме Stale Data (инвалидация кэша — одна из сложнейших проблем computer science). Пользователь может не увидеть только что загруженную аватарку.
- Ошибка при добавлении индексов в PostgreSQL на живой базе заблокирует таблицу (необходим `CREATE INDEX CONCURRENTLY`).

### Влияние на проект
Полная масштабируемость системы. Оптимальное использование ресурсов: кэширование 80% запросов экономит тысячи долларов на масштабировании API подов. Высокий показатель удержания пользователей (Retention).

### Примерная длительность
2 недели (Фокус на инфраструктуру и профилирование).

### Критерии завершения (DoD)
- [ ] В PostgreSQL созданы Covering Indexes для таблиц `Generation` и `Template` по полям `userId`, `createdAt` и `deletedAt`. Запросы типа `Get user images` используют `Index Scan` (проверяется через `EXPLAIN ANALYZE`).
- [ ] Подключен CDN (Cloudflare) с жестким кэшированием всей статики (шрифты, JS, CSS) и публичных сгенерированных изображений (`Cache-Control: public, max-age=31536000`).
- [ ] В NestJS API настроено Redis-кэширование (Interceptor) для высоконагруженных GET запросов (каталоги стилей, шаблоны). Инвалидация кэша происходит при изменении сущностей в админке.
- [ ] На фронтенде (Next.js) настроен `preload` критических ресурсов (шрифты).
- [ ] Анализ Web Vitals (LCP, CLS, FID) в Google Search Console подтверждает "зеленую" зону для мобильных устройств.


## 🧪 Sprint 5 — Testing & QA Automation

### Цель
Создать автоматизированный барьер качества (Quality Gate) в CI/CD пайплайне для предотвращения регрессий при частых деплоях (Continuous Deployment). 

### Ожидаемый результат
Невозможность случайно задеплоить сломанный код в Production. Все слои (DTO, Бизнес-логика, База Данных, UI) покрыты тестами. Инженеры могут уверенно релизить фичи по пятницам.

### Причины выполнения
Тестирование вручную (QA) на проекте такого масштаба невозможно — разработчики будут тратить 50% времени на регрессионное тестирование старого функционала. Баги в биллинге (ошибки при оплате) или авторизации слишком дороги для бизнеса. Отсутствие тестов парализует рефакторинг — инженеры боятся трогать старый код.

### Зависимости
Система должна быть полностью стабилизирована и типизирована (Sprints 1-4). Написание End-to-End тестов до фиксации UI-макетов и контрактов приведет к постоянной поломке тестов (Flaky tests).

### Риски
- Написание тестов на нестабильные внешние зависимости (OpenAI/Stripe) без грамотного мокирования сделает пайплайн нестабильным.
- Flaky тесты (ложноположительные или ложноотрицательные срабатывания E2E) приведут к тому, что разработчики начнут игнорировать результаты CI и просить отключить блокировку мержа.

### Влияние на проект
Радикальное улучшение Developer Experience (DX). Ускорение цикла разработки (Time-to-Market), так как команда тратит время на написание фич, а не на поиск багов. Бизнес защищен от критических поломок (остановка платежей).

### Примерная длительность
3 недели (Написание тестовой инфраструктуры и покрытие критического пути).

### Критерии завершения (DoD)
- [ ] В NestJS настроено тестовое окружение (Testcontainers) для поднятия реального PostgreSQL и Redis при запуске интеграционных тестов.
- [ ] Написаны Unit-тесты (Jest/Vitest) для всех DTO, Guard-ов (RolesGuard), и сложных бизнес-сервисов (начисление и списание кредитов).
- [ ] Интеграционные тесты (Supertest) покрывают 100% критических эндпоинтов API (Авторизация, Платежи, Запуск генерации).
- [ ] Настроен Playwright для E2E-тестирования. Покрыт главный пользовательский сценарий: (Регистрация -> Успешная покупка -> Генерация картинки -> Просмотр в галерее -> Скачивание).
- [ ] В GitHub Actions настроены строгие Branch Protection Rules: Merge Pull Request невозможен, если хотя бы один тест упал.


## 🏭 Sprint 6 — Production, Scaling & Observability

### Цель
Подготовка и шлифовка инфраструктуры к приему нагрузки до 1,000,000 MAU. Настройка полного мониторинга (Observability), оповещений (Alerting) и процедур Zero-Downtime деплоя.

### Ожидаемый результат
Система работает в полностью автономном режиме. При резком наплыве пользователей Kubernetes (HPA / KEDA) самостоятельно поднимает нужное количество подов API и Worker-ов, а после спада нагрузки — убивает их для экономии бюджета. Инженеры видят любые аномалии на графиках Grafana задолго до того, как пользователи начнут жаловаться в саппорт.

### Причины выполнения
Отсутствие Observability делает проект "черным ящиком": невозможно понять, почему генерация зависла, или на каком этапе отвалился платеж. Без автоскейлинга маркетинговая кампания (виральный пост в TikTok) приведет к жесткому даунтайму (Hug of Death) и потере рекламных бюджетов.

### Зависимости
Завершение абсолютно всех предыдущих спринтов. Инфраструктура должна быть разделена на сервисы (Sprint 1), типизирована (Sprint 2), оптимизирована (Sprint 4) и покрыта тестами (Sprint 5).

### Риски
Ошибки в конфигурациях Terraform или манифестах K8s (например, неправильно выставленные Memory Limits) могут привести к циклическим рестартам подов (OOMKilled, CrashLoopBackOff), полностью положив Production. Настройка сложных систем трейсинга может добавить накладные расходы на производительность.

### Влияние на проект
Полная Enterprise Grade готовность. SLA (Service Level Agreement) на уровне 99.99% uptime. Возможность привлекать корпоративных клиентов (B2B сегмент) благодаря гарантиям надежности.

### Примерная длительность
2-3 недели.

### Критерии завершения (DoD)
- [ ] Внедрен Distributed Tracing (Sentry или OpenTelemetry). Уникальный `TraceId` позволяет отследить путь запроса от клика в браузере до конкретного лога Воркера.
- [ ] Развернут Prometheus. В Grafana созданы дашборды по методологиям RED (Rate, Errors, Duration) для API и USE (Utilization, Saturation, Errors) для серверов.
- [ ] Настроен Kubernetes Event-driven Autoscaling (KEDA) для `worker-ai`. Новые поды поднимаются автоматически, когда длина очереди BullMQ превышает заданный порог.
- [ ] Настроены критические алерты в Telegram / Slack / Opsgenie: падение БД, рост 5xx ошибок, застревание очередей.
- [ ] CI/CD пайплайн выполняет деплой по стратегии Rolling Update (Zero Downtime). Пользовательские сессии не прерываются при релизе новых версий.
- [ ] Настроено ежедневное автоматическое резервное копирование PostgreSQL с выгрузкой в S3 (WAL-G / pg_dump).
- [ ] Инженерная команда успешно провела учения по Disaster Recovery: БД была полностью восстановлена из бэкапа на пустом сервере менее чем за 15 минут.
