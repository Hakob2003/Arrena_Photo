# 🚀 ПОЛНЫЙ ПРОДАКШЕН ЧЕК-ЛИСТ (100k - 1M Пользователей)

Этот документ является исчерпывающим руководством для подготовки проекта Arrena Photo к запуску на массовую аудиторию. Выполнение этих пунктов гарантирует, что система выдержит Хабра-эффект (Slashdot effect), атаки, скачки трафика и аппаратные сбои.

---

## 🏗️ 1. Инфраструктура и Балансировка (Infrastructure & Scaling)

Масштабирование от 1000 до 1,000,000 пользователей требует перехода от вертикального масштабирования (покупка более мощного сервера) к горизонтальному (добавление новых серверов).

### 1.1 Kubernetes / Docker Swarm

- [ ] **Оркестрация развернута:** Монолитный `docker-compose.yml` на одном сервере заменен на кластер (Kubernetes или Docker Swarm).
  - _Детали:_ Использование кластера позволяет балансировать нагрузку между нодами.
- [ ] **Horizontal Pod Autoscaler (HPA):** Настроен автоскейлинг для Backend API.
  - _Метрика:_ Если средняя загрузка CPU подов API превышает 70% в течение 2 минут, автоматически поднимаются новые поды.
- [ ] **Zero-Downtime Deploy:** Настроен Rolling Update.
  - _Детали:_ При деплое новой версии старые контейнеры не убиваются, пока новые не пройдут readinessProbe.

### 1.2 Балансировка Нагрузки (Load Balancing)

- [ ] **Ingress Controller (Traefik / Nginx Ingress / AWS ALB):** Установлен перед кластером.
  - _Детали:_ SSL терминация происходит на балансировщике. Трафик внутри кластера ходит по HTTP (снижение overhead'а).
- [ ] **Health Checks (Liveness/Readiness):** Балансировщик знает, жив ли конкретный инстанс NestJS.
  - _Детали:_ `/api/v1/health` возвращает 200 OK. Если возвращает 503 или таймаут, балансировщик перестает слать туда трафик.
- [ ] **Sticky Sessions (WebSockets):** Настроена липкость сессий для Socket.io, чтобы один клиент общался с одним подом (если не используется Redis Adapter).

### 1.3 Воркеры и Очереди (BullMQ / Redis)

- [ ] **Скейлинг Воркеров по длине очереди (KEDA):** Воркеры масштабируются не по CPU, а по количеству джобов в BullMQ.
  - _Детали:_ Если в очереди >50 картинок на генерацию, Kubernetes должен автоматически поднять еще 5 подов воркера.
- [ ] **Изоляция Redis:** Redis для очередей вынесен на отдельный выделенный сервер/кластер (не шарится с кэшем сессий фронтенда).
- [ ] **Обработка OOM в Воркерах:** Воркеры ограничены по памяти (Memory Limits). Если воркер крашится из-за тяжелого изображения, BullMQ должен вернуть задачу в очередь (Retry) и затем поместить в Dead Letter Queue.

### 1.4 Фронтенд (Edge Delivery)

- [ ] **Next.js развернут на Vercel или собственном Edge-кластере.**
  - _Детали:_ Если свой кластер, перед ним ОБЯЗАТЕЛЬНО должен стоять мощный CDN (Cloudflare).
- [ ] **Кэширование статики (Assets Caching):** Все `.js`, `.css`, шрифты и изображения отдаются со статусом `Cache-Control: public, max-age=31536000, immutable`.
- [ ] **Brotli / Gzip Compression:** Балансировщик сжимает текстовые ответы (JSON/HTML) перед отправкой клиенту.

## 🗄️ 2. Отказоустойчивость Базы Данных (Database Fault Tolerance)

База данных (PostgreSQL) — это самое узкое горлышко (bottleneck) любого приложения. Когда упадет БД, упадет весь проект.

### 2.1 Архитектура PostgreSQL

- [ ] **High Availability (HA) Кластер:** PostgreSQL работает в режиме Primary-Replica.
  - _Детали:_ Настроен инструмент (например, Patroni или AWS RDS Multi-AZ) для автоматического failover. Если Primary падает, Replica становится новым Primary за <30 секунд.
- [ ] **Read Replicas (Разделение чтения/записи):** В Prisma настроен `@prisma/extension-read-replicas`.
  - _Детали:_ Все GET-запросы маркетплейса и истории идут на Реплику, разгружая Мастер.
- [ ] **PgBouncer (Connection Pooling):** Установлен пулер соединений.
  - _Детали:_ При 50 подах API и 20 Воркерах будет создано 700+ прямых коннектов к БД, что убьет Postgres. PgBouncer держит коннекты открытыми и переиспользует их (Transaction mode).

### 2.2 Индексы и Оптимизация Запросов

- [ ] **Slow Query Log:** Включено логирование медленных запросов (`log_min_duration_statement = 1000`).
  - _Детали:_ Любой запрос дольше 1 секунды пишется в лог для анализа через `pg_badger`.
- [ ] **Покрывающие Индексы (Covering Indexes):** Все частые WHERE и ORDER BY покрыты B-Tree индексами.
  - _Проверка:_ Запустить `EXPLAIN ANALYZE` на самые частые запросы (лента генераций). Должно быть `Index Scan` или `Index Only Scan`, никакого `Seq Scan` (Sequential Scan).
- [ ] **GIN индексы для поиска:** Для полнотекстового поиска (FTS) по шаблонам или промптам используется GIN индекс, а не `LIKE '%...%'`.

### 2.3 Управление Данными (Data Lifecycle)

- [ ] **Партицирование таблиц (Table Partitioning):** Таблица `Generations` (которая будет расти по миллиону строк в месяц) разбита на партиции по дате (по месяцам).
- [ ] **Очистка мертвых строк (VACUUM):** Настроен агрессивный `autovacuum`.
  - _Детали:_ В таблицах с высокой частотой UPDATE (например, баланс юзера) мертвые строки могут раздуть таблицу (Table Bloat).
- [ ] **Ограничение размера JSONB:** Колонки с метаданными генерации не хранят гигабайты мусора. Введен жесткий лимит на уровень приложения.

### 2.4 Хранилище (MinIO / S3)

- [ ] **Миграция Base64 завершена:** В БД больше нет бинарных данных. Весь медиа-контент лежит в S3.
- [ ] **S3 Lifecycle Policies:** Включены правила автоудаления.
  - _Детали:_ "Сырые" входные картинки (временные файлы от юзера) удаляются через 7 дней.
- [ ] **CDN для медиа:** Перед S3-бакетом стоит CDN (Cloudflare), картинки отдаются пользователю из ближайшего к нему Edge-узла.

## 🔒 3. Безопасность и Compliance (Security)

При 1,000,000 пользователей проект становится мишенью для хакеров, спамеров и конкурентов. Цена ошибки — слив базы и репутационный крах.

### 3.1 Network Security & DDoS

- [ ] **Cloudflare WAF Включен:** Настроен Web Application Firewall.
  - _Детали:_ Блокируются известные эксплоиты (SQLi, XSS), анонимные прокси и Tor-ноды (если не нужно иное).
- [ ] **Rate Limiting (API Gateway / Redis):** Установлены жесткие лимиты на все эндпоинты.
  - _Пример:_ Не более 10 логинов в минуту с одного IP. Не более 5 попыток ввода OTP-кода.
- [ ] **Закрытые порты (VPC):** База данных (5432), Redis (6379) и Worker'ы физически недоступны из внешнего интернета.
  - _Проверка:_ Сканирование публичного IP через `nmap` показывает открытыми ТОЛЬКО порты 80 и 443.

### 3.2 Application Security (OWASP Top 10)

- [ ] **Валидация Ввода (DTOs):** Все входящие данные в NestJS строго валидируются через `class-validator` (whitelist: true). Никакого принятия лишних полей в payload'е.
- [ ] **Защита от SQL Инъекций:** Разработчикам запрещено использовать конкатенацию строк в `prisma.$queryRaw`.
- [ ] **CORS Настроен жестко:** `origin` в NestJS установлен строго на `['https://arrena.ai', 'https://www.arrena.ai']`, а не на `*`.
- [ ] **Скрыты Stack Traces в Prod:** Глобальный Exception Filter никогда не отдает наружу внутренние пути сервера при 500 ошибке.

### 3.3 Authentication & Authorization

- [ ] **Пароли:** Хэшируются через `bcrypt` (с factor=10 или выше) или `argon2`.
- [ ] **Сессии (JWT):** Токены хранятся в `HttpOnly, Secure, SameSite=Strict` Cookies (или используется Refresh/Access Token паттерн для мобилок).
- [ ] **Уязвимость BOLA (IDOR):** Устранены баги Broken Object Level Authorization.
  - _Проверка:_ Пользователь A не может удалить картинку Пользователя B, даже если знает ее UUID (в `WHERE` всегда передается `userId: req.user.id`).
- [ ] **Аудит доступов админки:** Все действия администраторов (бан юзера, начисление кредитов) пишутся в базу Audit Logs, доступную только C-level руководителям.

### 3.4 Утечка секретов

- [ ] **Секреты не в коде:** В репозитории нет `.env` файлов с реальными ключами. Все секреты внедряются через GitHub Secrets / AWS Secrets Manager / HashiCorp Vault.
- [ ] **Ротация ключей:** Настроен процесс быстрой замены JWT Secret, Stripe Secret и OpenAI API Key в случае их компрометации (без необходимости деплоя всего приложения).

## 🛟 4. Резервное Копирование (Disaster Recovery)

"Люди делятся на два типа: те, кто еще не делает бэкапы, и те, кто уже делает."

### 4.1 Backup Базы Данных (PostgreSQL)

- [ ] **Point-in-Time Recovery (PITR):** Настроено непрерывное архивирование WAL (Write-Ahead Logs) через `WAL-G` или `pgBackRest`.
  - _Зачем:_ Позволяет восстановить базу на ЛЮБУЮ секунду в прошлом (например, ровно за 5 минут до того, как стажер дропнул таблицу).
- [ ] **Ежедневные полные дампы (pg_dump):** Каждую ночь создается логический бэкап и выгружается в удаленное S3 хранилище (ОБЯЗАТЕЛЬНО в другом дата-центре!).
- [ ] **Мониторинг Бэкапов:** Настроен алерт (Cronitor/Healthchecks.io). Если скрипт бэкапа не отстучал "Успех" за 24 часа — тревога в Telegram/Slack.
- [ ] **Шифрование Бэкапов:** Дампы базы шифруются GPG ключом перед отправкой в облако (Compliance requirement).

### 4.2 Тестирование Восстановления (Recovery Drill)

- [ ] **Проведен тестовый Recovery:** Как минимум один раз разработчик взял чистый сервер и успешно развернул на нем проект с базой из вчерашнего бэкапа по инструкции.
  - _Без этого пункта все бэкапы Шредингера (вы не знаете, работают ли они)._

### 4.3 Среда "Одного Клипа" (Infrastructure as Code)

- [ ] **Terraform / Ansible / Helm:** Вся инфраструктура описана кодом.
  - _Зачем:_ Если ДЦ полностью сгорает, новая инфраструктура в другом облаке поднимается одной командой `terraform apply` за 15 минут.

## 📈 5. Мониторинг, Трейсинг и Алертинг (Observability)

Вы не можете чинить то, чего не видите. Мониторинг должен сообщать об инцидентах ДО того, как юзеры напишут в поддержку.

### 5.1 Сбор Метрик (Prometheus + Grafana)

- [ ] **Инфраструктура:** Мониторинг CPU, RAM, Disk I/O, Network для всех нод кластера (Node Exporter).
- [ ] **PostgreSQL:** Установлен `postgres_exporter`.
  - _Метрики:_ Connections count, Cache Hit Ratio (должен быть > 99%), Replication Lag, Deadlocks.
- [ ] **Redis / BullMQ:** Установлен `redis_exporter`.
  - _Метрики:_ Used Memory, Evicted Keys, Queue Length (самое важное для Воркеров).
- [ ] **Бизнес Метрики (NestJS Prometheus):**
  - Количество активных WebSocket соединений.
  - Скорость генерации (гистограмма времени ответа OpenAI).
  - Количество HTTP 500 ошибок в секунду (Error Rate).

### 5.2 Алертинг (Система тревог)

- [ ] **Настроены жесткие алерты (PagerDuty / Opsgenie / Telegram):**
  - 🚨 **CRITICAL (Звонок посреди ночи):** Сайт лежит (Blackbox exporter HTTP != 200), база отвалилась, кончилось место на диске.
  - ⚠️ **WARNING (Уведомление в Slack):** CPU > 85% дольше 10 минут, очередь BullMQ > 1000 задач, заканчивается баланс на счету OpenAI/Replicate (важно!).

### 5.3 Логирование (Centralized Logging)

- [ ] **Сбор логов (ELK / Loki / Datadog):** Логи со всех подов (API, Worker, Frontend) сливаются в единое хранилище. Разработчикам не нужно делать `kubectl logs` руками.
- [ ] **Структурированные логи (JSON):** NestJS переведен на Pino. Логи выводятся в формате JSON, что позволяет делать по ним аналитические выборки.
- [ ] **Correlation ID / Request ID:** Каждому HTTP запросу от браузера присваивается уникальный ID. Этот ID прокидывается в Воркер и Sentry, позволяя проследить весь жизненный цикл конкретного запроса юзера по логам всей распределенной системы.
- [ ] **PII Masking:** В логах скрыты пароли, номера карт и личные токены (Redact).

### 5.4 Отслеживание Ошибок (Sentry)

- [ ] **Sentry Backend:** Установлен в NestJS. Все Unhandled Exceptions улетают в дашборд.
- [ ] **Sentry Frontend:** Установлен в Next.js. Отлавливает ошибки рендера (ErrorBoundary) и сетевые сбои у клиентов.
- [ ] **Performance Tracing:** В Sentry включено отслеживание транзакций. Можно найти самый медленный SQL запрос или самый медленный API эндпоинт на реальных пользователях.
- [ ] **Source Maps:** В Next.js CI/CD настроена загрузка Source Maps в Sentry, чтобы вместо обфусцированного `a.b is undefined` видеть точную строчку исходного TypeScript кода.

## 🔍 6. SEO, Аналитика и Маркетинг (Growth Preparedness)

Продукт должен не только не падать, но и уметь масштабировать маркетинг и продажи.

### 6.1 Аналитика Продукта (Mixpanel / Amplitude / PostHog)

- [ ] **Воронка конверсии:** Настроены трекинг-события: `PAGE_VIEW -> SIGN_UP -> CLICK_GENERATE -> PAY_SUBSCRIPTION`.
- [ ] **Когортный анализ:** Бэкенд отправляет `userId` вместе с событиями (Server-Side Tracking), чтобы данные не блокировались AdBlock'ами.
- [ ] **Интеграция с Ads:** Установлен Facebook Pixel, Google Tag Manager. Бэкенд шлет Conversion API (CAPI) эвенты при успешной покупке.

### 6.2 SEO готовность (Search Engine Optimization)

- [ ] **Server-Side Rendering (SSR):** Публичные страницы (Marketplace, Public Profile) отдаются как полностью готовый HTML через Next.js App Router.
- [ ] **Мета-теги:** Настроены динамические `title`, `description` и `OpenGraph` картинки для каждой публичной генерации.
- [ ] **Sitemap.xml:** Генерируется автоматически или по Cron'у, содержит ссылки на все публичные шаблоны.
- [ ] **Robots.txt:** Закрыты служебные страницы (`/admin`, `/api`, `/settings`).
- [ ] **Core Web Vitals:** LCP (Largest Contentful Paint) < 2.5s. Устранен CLS (сдвиги верстки) с помощью плейсхолдеров и жестких размеров картинок `next/image`.
- [ ] **Canonical URL:** На всех страницах проставлен `<link rel="canonical">` во избежание дублирования контента в индексе Google.

### 6.3 Биллинг и Финансы (Stripe)

- [ ] **Идемпотентность (Idempotency):** При создании платежа (Stripe Intent) передается ключ `idempotencyKey` во избежание двойного списания денег.
- [ ] **Безопасность Вебхуков:** Вебхуки от Stripe валидируются через проверку подписи (`stripe.webhooks.constructEvent`) с использованием Raw Body.
- [ ] **Обработка неудач (Dunning):** Реализована логика отзыва Pro статуса, если у пользователя не списались деньги за следующий месяц.
